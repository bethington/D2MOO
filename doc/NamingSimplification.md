# Naming Simplification Proposal

Goal: drop redundant naming baggage (`D2` prefix, `Strc` suffix) across the codebase and
converge on the **Fortification (BH/slashdiablo) naming style** as the ruling standard for
types, functions, and fields. Where Fortification has a community-canonical name for a
struct, we adopt it verbatim; where it doesn't, we name things *the way Fortification
would* — bare PascalCase nouns, Hungarian camelCase fields, `MODULE_VerbPhrase` bindings.

This also aligns the C++ source with the Ghidra unified type vocabulary, which already
uses Fortification names as primary (see `conformance/fort_rename_map.json` and the
unify_types pipeline).

The complete machine-readable map (629 entries) is
`conformance/naming_simplification_map.json`, regenerated by
`conformance/tools/gen_naming_simplification_map.py`. Tier summary:

| Tier | Count | Meaning |
|---|---:|---|
| `fort_canonical` | 27 | adopt the exact Fortification/community name |
| `curated` | 9 | judgment call (collision or bad mechanical result) |
| `mechanical` | 364 | strip `D2`/`Strc`, name already good |
| `enum_strip` | 186 | strip `D2C_`/`D2` from enum type names |
| `needs_name` | 22 | `Unk*`/address-derived — needs a real name when understood |
| `keep` | 21 | already clean |

## 1. Type naming standard

- Bare **PascalCase**, no project prefix, no `Strc` suffix: `struct Inventory`, not
  `struct D2InventoryStrc`.
- Suffix vocabulary (Fortification conventions, already half-present in D2MOO):
  - `…Txt` — data-table row mirroring a `.txt` file (`ItemsTxt`, `MonStatsTxt`). The
    existing `D2*Txt` family just loses the `D2`.
  - `…Data` — per-unit payload (`ItemData`, `PlayerData`, `MonsterData`, `ObjectData`).
  - `…Node` / `…List` / `…Table` — intrusive links and containers.
- The `D2GSPacketClt##`/`D2GSPacketSrv##` wire structs keep their opcode-keyed names,
  minus the `D2`: `GSPacketClt01`, `GSPacketSrvB4`. The opcode *is* the meaning.

### Decision points (recommendation stated, alternative noted)

1. **`Room1` / `Room2`** — adopted per the Fortification-as-standard directive; every D2
   modder knows `pRoom1`/`pRoom2`, and Ghidra's unified vocabulary already uses them.
   The trade-off: D2MOO's current names (`D2ActiveRoomStrc` = collision/active room,
   `D2DrlgRoomStrc` = logical/persistent room) are *more descriptive* — and these types
   were already renamed once (`D2RoomStrc`/`D2RoomExStrc`). If descriptiveness is
   preferred, `ActiveRoom`/`DrlgRoom` are the fallback; but then the codebase diverges
   from every community source and from Ghidra.
2. **`ActMisc`** for `D2DrlgStrc` — a semantically weak name, but universally recognized
   (`Act.pMisc`). Same reasoning as Room1/Room2.
3. **`UnitAny`** vs plain `Unit` — `UnitAny` is the community term and communicates "this
   is every entity type, check `dwUnitType` before touching the union". Recommended.
4. **Generic-name flags** — `Effect`, `Field`, `Link`, `Property`, `Task`, `Tile`,
   `Control` are flagged `risky_generic` in the map: legal C++, no Win32 macro clash
   found, but grep-unfriendly. Accepted as-is; revisit individually if they hurt.
5. **`PacketList`** — `D2PacketListStrc` (D2Packet.h) collides with an existing *nested*
   `struct PacketList` in `GAME/Clients.h`. Legal (different scopes) but the nested one
   should be folded in or renamed during migration.

### `needs_name` (22 types)

`D2Unk*Strc*`, `D2WinImage2Strc`, `D2TimerArg2Strc`, and the two address-derived
`D2_6FD106F0_Strc`/`D2_6FD10880_Strc` get interim mechanical strips (`UnkMonCreate2`,
`Unk6FD106F0`…) but the real fix is a descriptive name once the structure is understood —
ideally resolved through the conformance/fun-doc pipeline the same way canonical field
names are resolved (never offset-derived names).

## 2. Enum naming

Strip `D2C_` / `D2` from enum type names: `D2C_UnitTypes` → `UnitTypes`,
`D2C_ItemQualities` → `ItemQualities`, `D2DrlgFlags` → `DrlgFlags`. Enum *members* keep
their existing screaming-case domain prefixes (`UNIT_PLAYER`, `COMP_HEAD`) — they match
community usage already. One curated fix: enum `D2Jungle` → `JungleIds` (the struct takes
`Jungle`). Also fix the `D2HarcodedConstants` typo → `HardcodedConstants` in passing.

## 3. Function naming

**No mass rename needed.** D2MOO's `COMPONENT_Action()` (`DUNGEON_GameToClientCoords`,
`ITEMS_GetItemType`) is structurally identical to Fortification's ruling
`MODULE_VerbPhrase` pattern (`D2COMMON_GetLevel`, `D2CLIENT_GetPlayerUnit`). The
component tags stay — they are finer-grained than Fort's DLL tags, which is a feature.

Cleanups to fold into ongoing work:
- Ordinal-named leftovers (`D2Common_11263`) get a `COMPONENT_VerbPhrase` name as they
  are reverse-engineered (existing policy; the conformance pipeline drives this).
- Plain C++ helpers that aren't exported game functions may use bare `VerbNoun()`
  PascalCase (Fortification's helper style) — no component tag required.
- Reverse-engineered `a1, a2` parameter names continue to be replaced with Hungarian
  camelCase names as functions are proven.

## 4. Field naming

Keep Hungarian camelCase (both projects agree). The standard prefix set, merged from
Fortification usage:

`dw` (u32) · `w` (u16) · `n` (int/index/count) · `b` (BOOL/byte) · `f` (flag bitfield) ·
`p`/`pp` (pointer levels) · `fp`/`fn` (function pointer) · `sz`/`wsz` (C string) ·
`t` (by-value nested struct) · `e` (enum-typed) · `cb` (byte count) · `h` (handle) ·
`x`/`y` (coordinates)

Deliberate deviations from Fortification, kept because they are better and already
load-bearing here:
- **Unknown fields stay `unk0xNN` / `pad0xNN` / `pUnused0xNN`** (offset-anchored), not
  Fort's `_1, _2` counters. The offset in the name plus the trailing `//0xNN` comment is
  what the conformance/struct-ledger tooling keys on.
- Every field keeps its trailing `//0xNN` offset comment (both projects do this; it stays
  mandatory).

Consistency fixes to apply during migration:
- **GUID handles**: fields typed `D2UnitGUID` are inconsistently `dwUnitId`,
  `dwOwnerGUID`, `dwOwnerId`. Standardize on `<owner>GUID` spelling with the `dw` prefix
  retained for layout honesty: `dwUnitGUID`, `dwOwnerGUID`.
- **By-value nested structs** use `t` consistently: `pSeed` (by-value!) → `tSeed`,
  `Stats` → `tStats`, matching the existing `tCoords`.
- `dw` on signed ints (`int32_t dwExpireFrame`) → `n` prefix when the type is signed.

## 5. Migration notes

- Blast radius is dominated by `UnitAny` (257 files), `Game` (194), `Room1` (139); the
  tail is shallow (most types touch < 10 files). Every old name is globally unique
  (`D2*Strc`), so renames are safe textual whole-word replacements — no semantic
  ambiguity. `statictests/` struct-layout asserts and the conformance regression suite
  gate each phase.
- Suggested phases: (1) enums + mechanical tail, (2) data-table `Txt`/`DataTbl` family,
  (3) the fort-canonical core (unit/room/stat cluster) in one commit so grep history
  stays clean, (4) field-consistency fixes, opportunistically.
- Update `CLAUDE.md` Code Conventions and fill the `doc/Coordinates.md` naming-convention
  TODO section when phase 1 lands; keep `naming_simplification_map.json` as the
  authoritative old→new ledger (Ghidra sync + conformance tooling can consume it).
